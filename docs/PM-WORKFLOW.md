# PM 工作流程：从想法到可执行的开发计划

> 一个人做项目时，你就是 PM。这份文档定义了"PM 阶段"要产出什么、怎么产出。

---

## PM 阶段的产出物清单

| # | 产出物 | 是否必须 | 工具 | 说明 |
|---|--------|---------|------|------|
| 1 | 产品 PRD | ✅ 必须 | PRD-TEMPLATE.md + AI 对话 | 需求的唯一来源 |
| 2 | 原型图 | 视情况 | Figma / 手绘 / AI 生成 | 复杂 UI 才需要 |
| 3 | API 契约 | ✅ 必须 | PRD 中的表格 | 前后端协作的桥梁 |
| 4 | 数据模型 | ✅ 必须 | PRD 中的表格 | 数据库设计的起点 |
| 5 | 开发排期 | ✅ 必须 | Plan Mode 生成 | 模块拆分+顺序 |

---

## Step 1：需求打磨（30-60 分钟）

### 1.1 初始脑暴
把你脑子里零散的想法全部写下来，不用在意格式。可以是：
- 一段语音转文字
- 一堆 bullet points
- 几张竞品截图

### 1.2 AI 辅助结构化
将上述内容喂给 AI（DeepSeek/Claude/ChatGPT 均可），使用以下 prompt：

```
我有一个项目想法，以下是我的初步描述：
---
[粘贴你的想法]
---

请帮我：
1. 梳理成结构化的需求点（按功能模块分组）
2. 指出描述中模糊或矛盾的地方
3. 生成一份"需求澄清问卷"，包含 10-15 个我需要回答的问题
4. 建议 MVP 应该包含哪些功能，哪些可以后续迭代
```

### 1.3 回答问卷 → 生成 PRD
回答完 AI 的问卷后，让它根据 PRD-TEMPLATE.md 格式生成完整 PRD。

---

## Step 2：原型设计（0-60 分钟，视复杂度）

### 什么时候需要原型？

| 情况 | 需要原型？ | 方式 |
|------|----------|------|
| API 服务/CLI 工具 | ❌ 不需要 | PRD 够了 |
| 简单 CRUD 后台 | ⚠️ 可选 | 文字描述 + wireframe |
| 面向用户的产品 | ✅ 需要 | Figma / v0.dev / 截图 |

### 原型的三个层级（选一个就行）

**Level 1 — 文字描述**（最快，适合简单页面）
```
登录页面：
- 居中卡片布局
- 邮箱输入框 + 密码输入框
- "登录" 按钮（主色调）
- 底部 "还没有账号？注册" 链接
```

**Level 2 — 快速线框图**（中等复杂度）
- 用 Figma / Excalidraw 画个大概布局
- 截图后用 @Image 喂给 Cursor

**Level 3 — 高保真原型**（产品级）
- Figma 完整设计
- 用 Builder.io 的 Figma → Code 插件
- 或截图喂给 Cursor 让它还原

---

## Step 3：技术方案 → 开发计划（在 Cursor 中完成）

PRD 完成后，在 Cursor 中用 Plan Mode：

```
请阅读 @docs/PRD-TEMPLATE.md（替换为你的实际 PRD 文件）

基于这份需求文档，帮我制定开发计划：
1. 项目目录结构（具体到文件路径）
2. 数据模型设计（SQL/ORM 级别）
3. API 路由设计（路径 + 方法 + 请求/响应）
4. 前端页面和组件拆分
5. 开发顺序（考虑依赖关系，先做什么后做什么）
6. 每个步骤的验收标准

只输出计划，等我确认后再开始写代码。
```

## 常见 PM 错误

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| 跳过 PM 阶段直接开写 | AI 瞎猜需求，返工多 | 至少花 30 分钟写 PRD |
| PRD 写得太抽象 | AI 生成的代码偏离预期 | 用用户故事+验收标准量化 |
| 不做 API 规划 | 前后端对接时才发现接口不匹配 | PRD 里就列出 API 清单 |
| 一口气写完所有需求 | 范围膨胀，永远做不完 | 严格区分 P0/P1/P2 |
